# OpenBrief v0.1.4

This patch release moves the brief delivery item count into SQLite-backed runtime configuration while preserving the default seven-item behavior for existing databases.

## Added

- Added the `max_delivery_items` runtime config key with a default of 7 and a valid range of 1 through 25.
- Added the `set_brief_options` runner config action for updating `max_delivery_items` through `openbrief config`.
- Added `max_delivery_items` to `run_brief` JSON so production agents can cap delivered candidate bullets without reading config separately.
- Added a targeted production-agent eval scenario that configures a non-default delivery limit, produces more candidates than the limit, records the exact delivered brief, and verifies clean runner-only hygiene.

## Changed

- Updated the OpenBrief skill to use `max_delivery_items` from the `run_brief` result, include all `must_include` items first, and fill only the remaining configured slots from `candidates`.
- Invalid stored `max_delivery_items` values now fall back to 7 and surface a runtime health warning instead of failing the brief.

## Verification

- Local release gates passed: release-doc validation, skill validation, formatting check, golangci-lint, unit tests, and the targeted configured-max-delivery-items production-agent eval.
